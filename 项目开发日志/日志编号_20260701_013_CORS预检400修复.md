### 开发内容详述

**问题现象：**
- 文件中心页面打开后，文件夹列表显示"Failed to fetch"
- 财务中心页面打开后，预算列表和订阅列表显示"获取失败"
- 密钥中心也出现类似问题

**根因分析（层级1）：OPTIONS 预检 400**
Nginx 日志：
```
GET  /api/v1/files/folders/  → 307 Temporary Redirect
OPTIONS /api/v1/files/folders → 400 Bad Request
```

浏览器向 `/files/folders/`（带尾斜杠）发 GET 请求，FastAPI 返回 307 重定向到 `/files/folders`（无尾斜杠）。浏览器收到重定向后，对目标 URL 发 CORS OPTIONS 预检，但预检返回 400。

**根因分析（层级2）：二层子路由特例**

只有三个端点受影响：`/files/folders`、`/finance/budgets`、`/finance/subscriptions`。它们是项目中**唯一的二层子路径 GET 端点**（路由定义如 `@router.get("/folders")` 位于 `prefix="/files"` 下）。

一层路由（如 `/tasks/`、`/contacts/`）不存在此问题——FastAPI CORS 中间件直接处理 OPTIONS，不触发重定向。

使用 OpenAPI 验证确认三层路由表：
- `/api/v1/files/folders`: post, get
- `/api/v1/finance/budgets`: get, post
- `/api/v1/finance/subscriptions`: get, post

**根因分析（层级3）：Nginx proxy_http_version**

直接访问后端（`curl localhost:8000/api/v1/files/folders -H "Origin:..."`）OPTIONS 返回 200。但经 Nginx 后返回 400。

原因：Nginx 默认 `proxy_http_version 1.0`，FastAPI CORS 中间件在 HTTP/1.0 下无法正确处理 OPTIONS 预检。添加 `proxy_http_version 1.1` 后 OPTIONS 预检正常返回 200 及完整 CORS 头。

**修复方案（双重修复）：**

1. **前端修复**（files.ts、budgets.ts、subscriptions.ts）：
   - `/files/folders/` → `/files/folders`（避免 307 重定向触发 CORS 预检失败）
   - `/finance/budgets/${query...}` → `/finance/budgets${query...}`（同样去掉尾斜杠）

2. **Nginx 修复**（shared.conf）：
   - 添加 `proxy_http_version 1.1` 确保 FastAPI CORS 中间件正常工作

### 遇到的问题与解决

- **问题1**：误判为全局 API 尾部斜杠问题，用 sed 批量修改了 19 个 API 文件，导致**全部** API 请求失败（FastAPI 一层路由需要尾部斜杠）。
  - 解决：`git checkout -- .` 回滚所有 API 文件，仅对三个二层子路由文件执行精确修复。
- **问题2**：Docker Compose 全量重启后问题依旧。
  - 解决：问题不在容器状态，而在 Nginx 的 `proxy_http_version` 和前端路径。
- **问题3**：密钥中心也报告失败，但实际 API 正常返回。
  - 排查：可能是浏览器缓存了 OPTIONS 预检失败的结果（`access-control-max-age: 600` 缓存 10 分钟），刷新后恢复正常。

### 重要经验教训

1. **二层子路由的 API 调用不要带尾部斜杠**。一层路由（如 `/tasks/`）可以保留尾部斜杠，FastAPI 正常处理。
2. **Nginx 代理 API 必须设置 `proxy_http_version 1.1`**，否则 FastAPI CORS 中间件可能无法正确响应 OPTIONS 预检。
3. **批量修改前先用 git 做还原点**，方便回滚。
4. **CORS 预检失败会被浏览器缓存**（`access-control-max-age`），排查时需考虑缓存因素。
