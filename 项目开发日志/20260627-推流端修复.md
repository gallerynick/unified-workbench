### 开发内容详述
对推流/播放全链路进行了端到端审查，涵盖前端推流端 (StreamStudio.tsx)、后端 WebSocket 中继 (stream_ws.py)、前端观看端 (StreamWatch.tsx) 及 Nginx 代理配置。审查确认：

- 后端 `stream_ws.py` 已正确使用 `receive_bytes()` 接收二进制 WebM chunks，并通过 `relay.relay()` 向所有 subscriber 转发。
- Nginx `/ws` location 的 WebSocket upgrade 代理配置正确。
- 观看端 `StreamWatch.tsx` 的 MediaSource、SourceBuffer、重连逻辑已通过多轮修复和 Oracle 验证。

但在推流端 `startStream()` 中发现一个关键隐患：`getStreamKey()` 返回新 key 后通过 `setStreamKey()` 更新 React state，但创建 WebSocket 时使用的仍是外层闭包中的旧 `streamKey` 值。如果 `loadSettings()` 尚未完成或服务端返回了更新的 key，推流端会连接到一个错误的（可能是空的）stream_key，导致数据发送到鬼魂房间，观看端永远收不到。

修复方式：在 `startStream()` 中引入局部变量 `currentStreamKey`，用 API 返回的最新 key 创建 WebSocket，确保推流端和观看端的 key 始终一致。

同时补充了推流端关键诊断日志（WS open/close/error、MediaRecorder mimeType、chunk 大小等），方便后续浏览器控制台排查。

### 遇到的问题与解决
- **问题**：用户报告 "即使使用 VLC 也无法拉流播放"。
  - **根因分析**：后端只有 WebSocket 流端点 (`/ws/stream/{key}`)，没有任何 RTMP、HLS、HTTP-FLV 或 DASH 输出端点。VLC 不支持 WebSocket 协议，因此无法直接拉流。系统的观看端使用浏览器内置的 `MediaSource` + `<video>` 来消费 WebSocket 传输的 WebM chunks，这是私有协议实现。
  - **说明**：这不是代码 bug，而是系统架构当前未实现标准流媒体输出。
- **问题**：推流端可能因异步 state 使用旧 key 导致数据发错房间。
  - **解决方案**：`startStream()` 使用 API 返回的 `keyRes.data.stream_key` 直接作为 WebSocket URL 的 key，不再依赖 React state 闭包。
- **遗留问题 / 待办**：
  - 系统未实现 HTTP/HLS/RTMP 流媒体输出，外部播放器（VLC、ffplay 等）无法直接播放。若未来需要，应增加转码/封装服务（如 SRS/nginx-rtmp/Node Media Server）或 HLS.js 播放方案。
  - 需要用户在推流端打开浏览器 Developer Tools → Console，观察 `[PUSH]` 开头的日志，确认 MediaRecorder 是否成功输出 chunk。

### 补充修复（14:30）— 编解码器声明不匹配导致 SourceBuffer 循环崩溃

用户提供观看端控制台日志后，发现了一个持续出现的循环崩溃模式：

```
appendBuffer ok 24850B          ← init segment 成功
SourceBuffer error event        ← 第二个 chunk 立即失败
创建新 MediaSource(...)          ← 自我恢复
appendBuffer ok 17906B          ← 又成功...
SourceBuffer error event        ← 又失败...
（无限循环）
```

#### 根因分析

推流端没有音频源时，`MediaRecorder` 的 MIME 类型是 `video/webm;codecs=vp8`（纯 VP8 视频）。但观看端的三级编解码器回退以 `vp8,opus` 开始：

```typescript
['video/webm; codecs="vp8,opus"', 'video/webm; codecs="vp8"', 'video/webm']
```

`addSourceBuffer('video/webm; codecs="vp8,opus"')` 在 Chrome 中成功（浏览器同时支持 VP8 和 Opus）。SourceBuffer 内部分配了 VP8 和 Opus 两个轨道缓冲区。但当 init segment 到达后，它只声明了 VP8 视频轨道——不存在 Opus 音频轨道。第二个包含 VP8 帧的 Cluster chunk 到达时，SourceBuffer 的 Opus 轨道缓冲区保持未初始化状态，导致 MSE 层抛出解码错误。

#### 修复

将回退顺序反转，使 `vp8` 排在第一位，与推流端实际编码匹配：

```typescript
['video/webm; codecs="vp8"', 'video/webm; codecs="vp8,opus"', 'video/webm']
```

同时修改了推流端的 key 异步问题并补充了诊断日志（见上文）。
