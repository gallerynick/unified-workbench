### 开发内容详述

**Bug 1 — 摄像头/屏幕源强制包含音频（StreamStudio.tsx:172, 176）**：
- **根因**：`getUserMedia({ video: true, audio: true })` 和 `getDisplayMedia({ video: true, audio: true })` 在添加视频源时同步请求了音频轨道。
- **修复**：改为 `audio: false`。音频需通过显式添加「麦克风」源。

**Bug 2 — 推流端未合并音频源轨道（StreamStudio.tsx startStream）**：
- **根因**：原代码 `streamRef.current = videoSource.stream!` 只使用视频源的流，音频源的 MediaStream 从未被传入 MediaRecorder，导致即使添加了麦克风源也无法推流音频。
- **修复**：新增合成流逻辑——从视频源提取视频轨道、从所有音频源提取音频轨道，合并到 `new MediaStream()` 后传给 MediaRecorder。

**Bug 3 — MediaRecorder mimeType 与实际流内容不匹配（StreamStudio.tsx）**：
- **根因**：固定使用 `video/webm;codecs=vp8,opus`，但 `audio: false` 后流中无音频轨道。Chrome 可能生成带空音频轨道的 WebM，导致观看端 SourceBuffer 编解码器不匹配。
- **修复**：根据 `audioTracks.length > 0` 动态选择 `vp8,opus` 或 `vp8`。

**Bug 4 — 后端 relay 滚动缓存丢失初始化段（stream_ws.py）— 致命**：
- **根因**：`room["buffer"]` 是 100 帧滚动窗口。推流持续超过 100 个 chunk 后，第一个 chunk（WebM 初始化段 = EBML + Segment + Tracks）被滚出。新订阅者收到的第一个 chunk 不含文件头，MediaSource 无法解码 → 永远黑屏/转圈。
- **修复**：新增 `init_segment` 字段，`relay()` 中保存第一个 chunk。`add_subscriber()` 检查初始化段是否已在 buffer 头部，若已滚出则前置。

**Bug 5 — `feed()` 无限重试死循环（StreamWatch.tsx）**：
- **根因**：`sb.appendBuffer(next)` 抛异常时 `catch {}` 未移除损坏 chunk，下一轮 flush 重试同一 chunk → 死循环。
- **修复**：catch 块添加 `q.shift()` 并输出 console.warn。

**Bug 6 — `onclose` 闭包过期（StreamWatch.tsx）**：
- **根因**：`if (status === 'playing')` 捕获组件挂载时的初始值 `'idle'`，永不更新。
- **修复**：引入 `statusRef`（useRef）跟踪实时状态。

**Bug 7 — 观看端 codec 硬编码 + 无诊断日志（StreamWatch.tsx）**：
- **修复**：`addSourceBuffer` 先尝试 `vp8,opus`（兼容有音频/无音频），失败回退 `vp8`。添加 sourceopen / addSourceBuffer / appendBuffer / updateend / play() 全链路 console.log 诊断。

### 遇到的问题与解决
- 问题：用户反馈观看端"一直转圈"，控制台无错误。
  解决方案：根因是后端 relay 滚动缓存丢失 WebM 初始化段。修复后新订阅者始终先收到初始化段。
- 问题：推流端 mimeType 固定 `vp8,opus` 但流无音频轨道。
  解决方案：动态检测音频轨道数选择 codec。
