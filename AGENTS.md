## 开发总则（强制）

1. **基定优先** — 开发前必须先读《项目基定/开发基准文档.md》，所有功能、设计、技术选型以它为准。
2. **版本管理** — 使用语义化版本号（如 `v1.0.0`），通过 Git tag 管理版本。不需要创建版本目录。
3. **日志强制** — 开发前读最近日志了解进度；开发后按 `项目基定/项目开发日志格式.md` 的格式更新日志。
4. **改动透明** — 日志必须列出所有新增、修改、删除的文件。
5. **遇疑即问** — 与基定冲突、版本不明、需求不清时，停止并询问，不得自行假设。
6. **自动部署** — 每次完成任务后自动重部署以便测试。

## 关键安全约束（不可违反）

- 密码：bcrypt 哈希，≥8位含字母+数字
- 密钥分区：AES-256-GCM 加密，强制私有
- 查看密钥值：二次输入登录密码验证
- 会话：JWT + 刷新机制 + 闲置超时登出
- 传输：全站 HTTPS
- 审计日志：只增不改不删

## 代码规范

- 后端：PEP8 + ruff + black + mypy
- 前端：ESLint + Prettier + 严格 TypeScript
- API：RESTful，统一响应 `{ "code": 0, "msg": "", "data": {} }`
- 所有 API 用 Pydantic 模型，自动生成 OpenAPI 文档
- **深浅色模式**：所有组件或功能（按钮、浮窗、选中框、互动组件、信息、图标、文字等）都必须适配深浅色模式

## 版本管理与应用标识文件

### 版本号规则

- 版本号格式：语义化版本（如 `0.1.0`）
- **不要主动更新版本号** — 只有当用户明确说"这个版本完成"或"进入下一版本"时才更新
- 版本号存储在 `项目文件/backend/app/version.py` 的 `__version__` 变量中

### 应用标识文件（`.unified-workbench`）

**用途**：防止自动更新系统误将其他应用的仓库当作本应用的更新源。

**何时更新**：
- 当用户确认版本完成或进入下一版本时，**必须同步更新** `.unified-workbench` 中的 `min_version` 字段
- 例如：版本从 `0.1.0` 升级到 `0.2.0` 时，标识文件中的 `min_version` 也要改为 `0.2.0`

**标识文件格式**：
```json
{
  "app_id": "unified-workbench",
  "min_version": "0.1.0"
}
```

**验证机制**（详见 `项目基定/应用标识文件基定.md`）：
1. 远程仓库必须存在 `.unified-workbench` 文件
2. 文件中 `app_id` 必须与后端 `APP_ID` 常量一致
3. 远程版本必须高于当前版本

**默认仓库地址**：`gallerynick/unified-workbench`
